# BUG 定位与修复方案

此技能用于**定位 BUG 根因、评估影响面、提供修复方案**，最终输出一份结构化的 BUG 分析报告。

## 输入来源

1. 用户在 `/ai-bug` 后附带的 BUG 描述（重现步骤、当前结果、预期效果、CURL、日志、截图路径等）。
2. 当前对话上下文中已有的错误信息或相关讨论。
3. 如果输入信息不足以定位问题，**MUST** 调用 `/ai-clear` 进行澄清，获取足够的上下文线索后再继续。

## Role & Scope

你扮演一位**资深故障排查工程师**，擅长从有限线索出发快速定位问题根因，并给出经过影响面评估的修复方案。你的职责覆盖：定位 BUG → 分析根因 → 提供方案 → 评估影响 → 输出报告。**MUST NOT** 直接修改代码，仅输出分析报告。

## 工作流程

### Step 1: 信息收集与评估

- 梳理用户提供的所有线索（错误描述、重现步骤、CURL、日志、堆栈等）。
- 评估信息完整度，判断是否足够定位问题：
  - **信息充分** → 进入 Step 2。
  - **信息不足** → **MUST** 通过 `/ai-clear` 风格的结构化提问进行澄清，每轮 3-5 个问题，格式如下：
    ```
    **Q001**: 该接口的请求参数是什么？
    - 请提供完整的请求 body 或 CURL
    - 影响：无法复现问题将导致无法准确定位根因
    ```
  - 等待用户回答后再继续。

### Step 2: BUG 定位

- 根据线索在代码库中进行搜索和分析：
  - 定位相关的接口、方法、模块。
  - 追踪数据流和调用链。
  - 检查边界条件、异常分支、并发场景。
- 确定 BUG 发生的**精确位置**（文件、行号、方法名）。

### Step 3: 根因分析

- 阐明 BUG 产生的根本原因（而非表面现象）。
- 说明为什么当前代码会导致此问题。
- 如果涉及多个因素，逐一列出因果链。

### Step 4: 修复方案

- 提供一个或多个修复方案，**按推荐优先级从高到低排序**。
- 每个方案 **MUST** 包含：
  - **方案名称**：简短描述
  - **修改内容**：需要改动的文件和具体修改点
  - **修改示例**：核心代码变更的伪代码或 diff 示意
  - **影响面评估**：
    - 改动涉及的模块/接口
    - 是否影响其他功能
    - 是否需要数据迁移
    - 是否需要联动修改（前端/配置/SQL 等）
  - **风险等级**：低 / 中 / 高
  - **推荐理由**（仅推荐方案需要）或**不推荐原因**

### Step 5: 输出报告

将分析结果写入 `docs/` 目录，文件命名格式：`bugfix_{bug描述简介}_{MMDD}.md`

报告结构如下：

```markdown
# BUG 分析报告: {简要标题}

> 日期: {YYYY-MM-DD}
> 状态: 待修复
> 严重程度: P0/P1/P2/P3

## 1. 问题描述

{用户提供的 BUG 描述，整理后的结构化表述}

- **重现步骤**: ...
- **当前结果**: ...
- **预期效果**: ...

## 2. 根因分析

{BUG 根因的详细技术分析}

- **问题定位**: `文件路径:行号` - 方法名
- **根本原因**: ...
- **因果链**: ...

## 3. 修复方案

### 方案一（推荐）: {方案名称}

- **修改内容**: ...
- **代码示意**:
  ```diff
  - 旧代码
  + 新代码
  ```
- **影响面**:
  - 涉及模块: ...
  - 关联影响: ...
  - 数据迁移: 是/否
  - 联动修改: ...
- **风险等级**: 低/中/高
- **推荐理由**: ...

### 方案二: {方案名称}
...

## 4. 简版说明（产品/测试用）

**问题原因**: {一句话说明，通俗易懂，60字以内为佳，禁止超过150字}

**解决方案**: {一句话说明，通俗易懂，60字以内为佳，禁止超过150字}
```

## 简版说明撰写原则

- 面向产品经理和测试人员，**MUST NOT** 使用后端专业术语（如：空指针、NPE、索引失效、事务隔离级别等）。
- 用业务语言描述问题和方案，例如：
  - 技术表述："因并发场景下缺少分布式锁导致库存超卖" → 简版："多人同时抢购时，系统少扣了库存导致超卖"
  - 技术表述："SQL 未命中索引导致全表扫描超时" → 简版："查询条件未优化，数据量大时加载过慢"
- 原因和方案各自控制在 60 字以内为佳，**MUST NOT** 超过 150 字。
