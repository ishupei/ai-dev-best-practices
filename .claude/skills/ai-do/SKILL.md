# 任务执行引擎

此技能用于**依次执行**已拆解好的开发任务，逐个完成编码实现并验证，确保每一步都产出正确、可运行的代码。

## Usage

- 当用户输入以 `/ai-do` 开头时，**MUST** 触发此技能。
- 输入来源（按优先级）：
  1. 用户在 `/ai-do` 后指定了具体任务编号（如 `/ai-do Task 2.3`）→ 从该任务开始执行。
  2. 当前对话上下文中已有任务清单或前序 skill 的输出 → 直接基于上下文执行。
  3. 在 `docs/` 下**递归**查找 `TASKS-*.md`，并读取**同一功能目录**下的 `TDD-*.md`、`SRS-*.md`（`docs/<功能目录名>/`）。
- 如果以上均无可用输入，**MUST** 提示用户先提供任务清单或执行 `/ai-task`。

## Role & Scope

你扮演一位**高级开发工程师**，严格按照任务清单和技术设计进行编码实现。你需要编写高质量、可维护的代码，并在每个任务完成后进行验证。

## 工作流程

### 执行循环（逐任务执行）

对每个待执行的任务，**MUST** 按以下步骤执行：

#### 1. 任务确认
- 展示当前要执行的任务信息（编号、标题、描述）。
- 展示当前进度（第 X/Y 个任务）。
- 检查该任务的依赖是否已完成。
- 询问用户是否开始执行（如果用户未要求自动模式）。

#### 2. 上下文准备
- 读取任务描述中引用的设计文档章节。
- 读取任务涉及的现有文件（如果是修改已有文件）。
- 了解当前代码库的相关上下文（项目结构、命名约定、现有模式）。

#### 3. 编码实现
- 严格按照技术设计文档中的接口定义、数据模型、业务逻辑进行实现。
- 遵循项目现有的编码规范和风格（参考 `.cursor/rules/` 中的规则）。
- 实现要点：
  - **代码质量**：清晰、简洁、可维护
  - **错误处理**：合理的异常处理和边界检查
  - **命名规范**：与项目现有风格保持一致
  - **最小改动**：只做任务要求的事，不过度设计

#### 4. 自我验证
- 按照任务中定义的验证方式逐项检查。
- 运行相关测试（如果有）。
- 检查代码是否与技术设计一致。
- 检查是否引入了新的问题或与已完成任务产生冲突。

#### 5. 任务完成报告
- 输出本次任务的完成摘要：
  ```
  ✅ Task X.X: <任务标题> 已完成

  📝 变更内容：
  - 新建/修改了哪些文件
  - 核心实现要点

  ✔️ 验证结果：
  - [x] 验证项1 - 通过
  - [x] 验证项2 - 通过

  📊 进度：X/Y 任务完成（Phase N）

  ➡️ 下一个任务：Task X.X <标题>
  ```
- 等待用户确认后继续下一个任务。

### 异常处理

- **依赖未完成**：提示用户该任务依赖的前置任务尚未完成，建议先完成依赖任务。
- **设计不明确**：如果技术设计对某个实现细节不够清晰，向用户说明并提出你的建议方案，等待确认。
- **实现冲突**：如果当前任务的实现与已完成的代码产生冲突，分析原因并提出解决方案。
- **验证失败**：如果验证不通过，分析原因，修复后重新验证。

### 进度追踪

- 在 TASKS 文档中更新已完成任务的状态。
- 在任务标题前添加完成标记：
  - `✅` 已完成
  - `🔄` 进行中
  - `⏳` 待执行
  - `❌` 阻塞

## 自动模式

如果用户输入 `/ai-do auto`，则进入自动执行模式：
- 不需要每个任务前确认，连续执行所有任务。
- 但在遇到以下情况时 **MUST** 暂停并询问用户：
  - 设计不明确需要决策
  - 验证失败需要确认修复方案
  - 与已有代码产生冲突
