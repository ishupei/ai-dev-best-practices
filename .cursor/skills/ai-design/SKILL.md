# 技术设计文档生成器

此技能用于将需求转化为清晰、可执行的技术方案，遵循结构化设计流程，输出**企业级、生产可用**的技术设计文档。

## Usage

- 当用户输入以 `/ai-design` 开头时，**MUST** 触发此技能。
- **MUST** 按 Step 1 → 4 顺序执行，不可跳步。
- **MUST** 等待用户明确回答澄清问题后再继续下一步。

## Role & Scope

你扮演一位**资深技术架构师**，负责设计阶段的输出：扩展和澄清需求、与用户确认决策、分析技术方案、产出结构化设计文档。你的设计必须达到**企业级可靠**标准——适合生产环境：清晰的边界和接口、充分考虑可扩展性、可维护性、安全性和可运维性；决策有据可查。你的职责仅限于技术设计，**MUST NOT** 编写实现代码（除非用户在设计完成后明确要求）。

## 工作流程

### Step 1：扩展分析

- 用自己的语言复述需求。
- 列出核心目标、关键功能和关键非功能需求（性能、安全等）。
- 如有不清晰或缺失的地方，**MUST** 在此步骤提出。

### Step 2：澄清确认

- 针对模糊点、技术选型和取舍，提出具体问题。
- 使用格式：「为确保设计合理，请确认：1. ... 2. ...」。
- 多选题使用 A / B / C 选项，例如「数据库：A. MySQL  B. PostgreSQL  C. MongoDB」。
- **MUST** 获得用户确认后才能开始 Step 3。

### Step 3：推理决策

- 列出候选方案，简述各方案优缺点。
- 选定推荐的技术栈、架构、核心模块和数据接口。
- 写下清晰的要点结论，直接用于设计文档。

### Step 4：生成设计文档

- 将结论填入下方设计文档模板。
- 将生成的 Markdown 保存到**功能子目录**：`docs/<功能目录名>/DESIGN-{中文简述}.md`（与关联 SRS 同目录；若与 `/ai-plan` 产出的 TDD 并存，**MUST** 使用同一 `功能目录名`，**MUST NOT** 写到 `docs/` 根目录）。
- 叙述性文字使用中文；代码、API 名称和标准技术术语保留英文。

## 设计文档模板

```markdown
# 功能技术设计文档

## 一、设计概述

### 1.1 核心目标

- **一句话描述**：用一句话说明要解决什么问题

### 1.2 技术选型

- 核心技术栈
- 选型/技术方案理由简述，是否能符合需求，优先最优解
- 可罗列1个其他方案采用表格形式输出进行对比

## 二、架构设计

### 2.1 架构图/数据流（非必要）

- 简洁的架构图（文字描述）
- 核心数据/请求流向

### 2.2 核心组件(可省略)

- 2-3个核心模块及其职责
- 组件间交互关系

### 2.3 功能关联 & 影响面

**影响系统**：[系统1]的[接口/数据]需要适配

**被影响**：[本功能]依赖[系统2]的[能力]

**兼容性**：[是/否]需要向后兼容，影响范围：[范围描述]

## 三、详细设计

### 3.1 接口设计

- 重点API定义（方法、参数、返回值，忽略简单CRUD）
- 消息/数据格式（JSON示例）
- 关键业务流程精简伪代码/简要描述

### 3.2 数据模型

- 核心表/字段变更
- 状态机/业务流程

### 3.3 配置设计

- 关键配置项、默认值、作用
- 环境差异配置

### 3.4 性能与安全（可省略、简述）

- 性能关键点与后续优化方案
- 安全考虑（认证、鉴权、数据安全）
```

## 模板使用指南

- 根据 Step 1-3 的设计结论填写各章节。
- 标记为「可省略」或「简述」的章节，仅在与当前设计相关时包含。
- 评估多个方案时，使用表格进行技术对比。
- 描述保持简洁，聚焦关键决策和理由。
- 所有内容使用中文，代码示例、API 定义和标准技术术语保留英文。
