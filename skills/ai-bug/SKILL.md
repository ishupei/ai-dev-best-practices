# BUG 定位与修复方案

此技能用于**定位 BUG 根因、评估影响面、提供修复方案**，最终输出一份结构化的 BUG 分析报告。

## Usage

- 当用户输入以 `/ai-bug` 开头时，**MUST** 触发此技能。
- 用户可在 `/ai-bug` 后附带 BUG 描述（重现步骤、当前结果、预期效果、CURL、日志、截图等）。
- 输入来源（按优先级）：
  1. 用户在 `/ai-bug` 后直接提供的 BUG 描述信息。
  2. 当前对话上下文中已有的错误信息或相关讨论。
- 若判定**上下文不足以定位或复现**（见下文「与 ai-clear 的衔接」），**MUST** 在同一对话中**自动按 ai-clear 技能执行澄清流程**（读取并遵循 `skills/ai-clear/SKILL.md` 的 **Step 2** 规范），**MUST NOT** 仅用零散追问替代。

## Role & Scope

你扮演一位**资深故障排查工程师**，擅长从有限线索出发快速定位问题根因，并给出经过影响面评估的修复方案。你的职责覆盖：定位 BUG → 分析根因 → 提供方案 → 评估影响 → 输出报告。**MUST NOT** 直接修改代码，仅输出分析报告。

## 与 ai-clear 的衔接（上下文不足时自动澄清）

「自动调用」指：在同一轮对话中由你**读取并落实** `ai-clear` 的澄清规范，而非要求用户手动再输入 `/ai-clear`（用户若主动输入 `/ai-clear` 亦可）。

### 触发条件（满足任一即进入澄清阶段）

- 无法复现或无法缩小范围（缺环境、版本、分支、请求参数/报文、测试数据等）。
- 缺关键日志、堆栈、接口路径、或现象与错误信息对不上。
- 「当前结果 / 预期结果」缺失或语义模糊。
- 可能涉及多模块/多入口，但线索不足以选择排查路径。

### 澄清阶段 MUST 遵循（对齐 ai-clear Step 2）

- **MUST** 读取 `skills/ai-clear/SKILL.md`，按其 **Step 2：结构化提问** 执行：每轮 **3-5** 题、**Q001…** 编号、**A/B/C 且推荐项固定为 A**、**Asker 式**（裸露列表、快捷回复 `Q001-A`、本轮汇总 `Q001:A,Q002:B`）、📍 影响行**按需**（仅当选项会改变排查范围或验收边界时）。
- **MUST** 在首次进入澄清时用简短说明告知用户，例如：`当前 BUG 线索不足，已按 **ai-clear** 的澄清规范提问（仅复用提问形式，不更新 DRAFT）。`
- **MUST NOT** 用三反引号代码块包裹选项列表（与 ai-clear 一致）。
- **MUST NOT** 执行 ai-clear 的 **Step 1（审阅 DRAFT）** 或 **Step 3（更新 DRAFT 文档）**；澄清对象是 BUG 上下文，不是需求规格书。若用户明确要求同时澄清 DRAFT，再另按 `/ai-clear` 处理。
- **MUST NOT** 在关键信息仍缺失时强行给出确定根因或写入 `bugfix_*.md` 报告（可先输出「待补充信息」清单）。

### 澄清结束后

- 合并用户答复，更新心智中的 BUG 上下文，**继续本技能 Step 2（BUG 定位）及后续步骤**。

## 工作流程

### Step 1: 信息收集与评估

- 梳理用户提供的所有线索（错误描述、重现步骤、CURL、日志、堆栈等）。
- 评估信息完整度，判断是否足够定位问题：
  - **信息充分** → 进入 Step 2。
  - **信息不足** → **MUST** 进入上文「与 ai-clear 的衔接」澄清阶段，**禁止**使用与 ai-clear Step 2 不一致的随意格式。
- 等待用户回答后再继续。

### Step 2: BUG 定位

- 根据线索在代码库中进行搜索和分析：
  - 定位相关的接口、方法、模块。
  - 追踪数据流和调用链。
  - 检查边界条件、异常分支、并发场景。
- 确定 BUG 发生的**精确位置**（文件、行号、方法名）。

### Step 3: 根因分析

- 阐明 BUG 产生的根本原因（而非表面现象）。
- 说明为什么当前代码会导致此问题。
- 如果涉及多个因素，逐一列出因果链。

### Step 4: 修复方案

- 提供一个或多个修复方案，**按推荐优先级从高到低排序**。
- 每个方案 **MUST** 包含：
  - **方案名称**：简短描述
  - **修改内容**：需要改动的文件和具体修改点
  - **修改示例**：核心代码变更的伪代码或 diff 示意
  - **影响面评估**：
    - 改动涉及的模块/接口
    - 是否影响其他功能
    - 是否需要数据迁移
    - 是否需要联动修改（前端/配置/SQL 等）
  - **风险等级**：低 / 中 / 高
  - **推荐理由**（仅推荐方案需要）或**不推荐原因**

### Step 5: 输出报告

- **目录**：**MUST** 将报告写入 **`docs/bugfix/`** 目录（若不存在则创建）；**MUST NOT** 将 BUG 分析报告放在任何 `docs/<功能目录名>/` 功能子目录内，也 **MUST NOT** 放在 `docs/` 根目录。
- **路径示例**：`docs/bugfix/bugfix_{中文简述}_{MMDD}.md`
- **文件名**：`bugfix_{中文简述}_{MMDD}.md`（`中文简述` 为中文；`bugfix` 为文件名固定英文前缀）。

报告结构如下：

```markdown
# BUG 分析报告: {简要标题}

> 日期: {YYYY-MM-DD}
> 状态: 待修复
> 严重程度: P0/P1/P2/P3

## 1. 问题描述

{用户提供的 BUG 描述，整理后的结构化表述}

- **重现步骤**: ...
- **当前结果**: ...
- **预期效果**: ...

## 2. 根因分析

{BUG 根因的详细技术分析}

- **问题定位**: `文件路径:行号` - 方法名
- **根本原因**: ...
- **因果链**: ...

## 3. 修复方案

### 方案一（推荐）: {方案名称}

- **修改内容**: ...
- **代码示意**:
  ```diff
  - 旧代码
  + 新代码
  ```
- **影响面**:
  - 涉及模块: ...
  - 关联影响: ...
  - 数据迁移: 是/否
  - 联动修改: ...
- **风险等级**: 低/中/高
- **推荐理由**: ...

### 方案二: {方案名称}
...

## 4. 简版说明（产品/测试用）

**问题原因**: {一句话说明，通俗易懂，60字以内为佳，禁止超过150字}

**解决方案**: {一句话说明，通俗易懂，60字以内为佳，禁止超过150字}
```

## 简版说明撰写原则

- 面向产品经理和测试人员，**MUST NOT** 使用后端专业术语（如：空指针、NPE、索引失效、事务隔离级别等）。
- 用业务语言描述问题和方案，例如：
  - 技术表述："因并发场景下缺少分布式锁导致库存超卖" → 简版："多人同时抢购时，系统少扣了库存导致超卖"
  - 技术表述："SQL 未命中索引导致全表扫描超时" → 简版："查询条件未优化，数据量大时加载过慢"
- 原因和方案各自控制在 60 字以内为佳，**MUST NOT** 超过 150 字。
