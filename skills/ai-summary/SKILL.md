# 上下文摘要与进度快照

此技能用于把**当前对话中的有效信息**与 **`docs/<功能目录名>/` 内已有 SRS、TDD/DESIGN、TASKS 等** 对照整合，产出一份**短而准**的摘要文档，降低新开窗口或长对话导致的理解偏差，便于 AI 与用户快速对齐「需求是什么、已澄清什么、做到哪一步、下一步是什么」。

## Usage

- 当用户输入以 `/ai-summary` 开头时，**MUST** 触发此技能。
- 用户可在 `/ai-summary` 后指定**功能目录路径**或 **SUMMARY 目标文件名**；未指定时按下方优先级解析输入来源。
- 输入来源（按优先级）：
  1. 用户在命令后指定了 `docs/<功能目录名>/` 或其中某文件 → 以该目录为**唯一**写入与关联范围。
  2. 当前对话已明确在讨论某一功能 → 推导 `功能目录名` 并仅针对该目录读写。
  3. 在 `docs/` 下查找**最近有活动**（或最近修改）的 `SUMMARY-*.md` / 同目录下 `SRS-*.md` 所在目录作为默认范围。
- 若无法确定功能目录，**MUST** 请用户指定 `docs/<功能目录名>/`，**MUST NOT** 把多个需求的摘要混写入同一文件。

## Role & Scope

你扮演一位**技术文档编辑 + 项目经理（摘要向）**：只写**可核对**的事实与结论，不扩写需求。你的职责是**读**已有 Markdown 与对话、**写/更新** SUMMARY；**MUST NOT** 实现代码、**MUST NOT** 替代 SRS/TDD 的正文（摘要只指向与摘录）。

## 核心原则（精准优先）

- **可核对**：每条「已澄清结论」「范围边界」**MUST** 能在 SRS/TDD/TASKS 或本对话中的明确陈述中找到依据；无依据则列入 **待确认**，**MUST NOT** 写成既定事实。
- **不堆砌**：拒绝把长对话全文搬进 SUMMARY；只保留对后续决策与实现**有影响**的要点。
- **可冷启动**：新开窗口只读 `SUMMARY-*.md`（配合 SRS/TDD 标题与路径）应能恢复**80% 以上**协作语境；其余细节回链原文档。
- **版本感**：更新时调整 **摘要版本** 与 **更新日期**；若与上一版差异大，在 **变更要点** 中写 3～5 条 bullet。

## 产出位置与命名

- **默认路径**：`docs/<功能目录名>/SUMMARY-{中文简述}.md`（与同目录 SRS、TDD、TASKS 一致；**MUST NOT** 放在 `docs/` 根目录）。
- **文件名**：英文前缀 `SUMMARY-` + **中文简述**（可与功能目录名一致或更短）。
- **与 DESIGN/TDD 的关系**：摘要**不替代**设计文档；若本目录存在 `TDD-*.md` / `DESIGN-*.md`，SUMMARY 中 **MUST** 用「设计要点摘录 + 文件引用」指向原文，**MUST NOT** 复制大段接口/SQL。
- **可选：合并进 DESIGN**（仅当用户**显式要求**）：在 `DESIGN-*.md` **文末**追加 **「附录：上下文摘要与进度」**，按下方**输出模板**的同结构填写；**MUST NOT** 删除或覆盖原有设计章节。仍建议同时或定期维护独立 `SUMMARY-*.md`，便于单独 `@` 引用。

## 工作流程

### Step 1：锁定范围与读取

- 确定 `docs/<功能目录名>/`，列出该目录下 `SRS-*.md`、`TDD-*.md`、`DESIGN-*.md`、`TASKS-*.md`（若存在）。
- 读取上述文件（优先读与用户当前目标相关的章节）。
- 梳理当前对话中**用户明确确认**的决策、范围变更、里程碑、阻塞项（无文档支撑的对话结论标注为「对话确认」）。

### Step 2：提炼结构

- 将信息归入下方**模板章节**；无法归入且重要的，放入 **附录：杂项** 并标来源。
- 对已澄清与未澄清分开：**已澄清**须有 SRS/TDD/TASKS/对话依据；**未澄清**单独列表，**MUST NOT** 猜测补全。

### Step 3：写入或覆盖更新

- **新建**：若不存在 `SUMMARY-*.md`，按模板创建。
- **更新**：若已存在，合并新旧内容：保留仍有效的历史里程碑，更新「当前进度」「下一步」；冲突时**以文档与用户最新口述为准**，并在变更要点中说明。
- 在对话中输出 3～6 句**说明**：本次摘要覆盖范围、写入路径、与上一版主要差异（若有）。

## 输出模板

```markdown
# 上下文摘要：{功能或需求简称}

> **摘要版本**：v1.0
> **更新日期**：YYYY-MM-DD
> **功能目录**：`docs/<功能目录名>/`
> **关联文档**：SRS-xxx.md | TDD/DESIGN-xxx.md | TASKS-xxx.md（按实际存在列出）

## 1. 一句话目标

{用一句话说明当前需求/项目要达成什么，需与 SRS 一致或可追溯到 SRS}

## 2. 需求与范围快照

- **In Scope**：{要点列表，可标 F001}
- **Out of Scope**（若已约定）：{要点}
- **已澄清结论**（须有依据）：
  - {结论} — **依据**：SRS/TDD §x 或 对话确认于 YYYY-MM-DD
- **仍待确认**：{仅列问题，不写臆测答案}

## 3. 核心功能点

| 编号 | 功能点 | 状态 | 备注 |
|------|--------|------|------|
| F001 | … | 已澄清/待澄清 | … |

## 4. 设计与约束摘录

- **架构/模块**：{1～3 条要点，细节见 `TDD-*.md` / `DESIGN-*.md` 对应章节}
- **关键接口/数据**（仅列名称级）：{可选}
- **非功能约束**（安全、性能等）：{可选，摘录}

## 5. 当前进度与对话焦点

- **阶段**：{如：需求已澄清 / 设计中 / 开发中 / 联调}
- **本轮对话目标**（若适用）：{用户在本窗口 stated 的目标}
- **已完成**：{任务或文档状态，与 TASKS 勾选项对齐更佳}
- **进行中**：{…}
- **阻塞/风险**：{无则写「无」}

## 6. 里程碑与下一步

| 里程碑 | 目标日期/条件 | 状态 |
|--------|----------------|------|
| M1 | … | 未开始/进行中/完成 |

- **下一步（可执行）**：{1～3 条，动词开头}

## 7. 变更要点（相对上一版摘要）

- {仅更新时填写；首版可删本节或写「首版」}

## 8. 使用说明（给新开窗口的 AI）

- 请先阅读本文件，再按需打开 **关联文档** 中的 SRS/TDD/TASKS 对应章节；勿仅凭摘要做架构级变更。
- 若摘要与 SRS/TDD 冲突，**以 SRS/TDD 正文为准**，并应触发更新本 SUMMARY。
```

## 与 Cursor / 新开窗口的配合

- 用户在新会话中可 **@** `docs/<功能目录名>/SUMMARY-*.md` 快速加载；你应在 SUMMARY 中保持 **关联文档路径** 准确、**阶段与下一步** 最新。
- **MUST NOT** 在 SUMMARY 中保存密钥、token、个人隐私；**MUST NOT** 写入与项目无关的闲聊。

## Checklist

- [ ] 用户输入以 `/ai-summary` 开头，已触发本技能
- [ ] 已锁定单一 `docs/<功能目录名>/`，未混多需求
- [ ] 已阅读同目录 SRS/TDD/DESIGN/TASKS（按存在情况）
- [ ] 摘要内容可核对，臆测已标为待确认或未写入
- [ ] 已写入或更新 `docs/<功能目录名>/SUMMARY-{中文简述}.md`（或按用户要求仅在 `DESIGN-*.md` 文末追加附录）
- [ ] 已包含目标、核心功能点、已澄清项、进度/里程碑/下一步（按适用性）
- [ ] 对话中已说明路径与相对上一版差异（若有）
