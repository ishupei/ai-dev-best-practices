# /ai-do 代码规范约束

本文件定义 `/ai-do` 在执行开发任务时必须遵循的代码规范约束。只要触发 `/ai-do`，就 **MUST** 在编码前读取本文件，并在实现与验证阶段持续对照执行。

## 1. 适用范围

- 适用于 `/ai-do` 触发后的所有代码生成、代码修改、测试补充、配置变更与文档落地。
- 适用于新建文件与修改现有文件。
- 若任务只要求分析、不允许改代码，则本规范仅用于约束输出的实现建议与修复方案。

## 2. 规则优先级

出现多份规范时，按以下优先级执行，低优先级不得覆盖高优先级：

1. 用户本轮明确指令
2. TASK / DESIGN / PLAN / DRAFT 文档中的硬性约束
3. 当前技能文件 `skills/ai-do/SKILL.md`
4. 本规范文件 `skills/ai-do/references/code-standards.md`
5. 目标代码目录中的现有编码风格、命名方式、分层模式
6. 通用最佳实践

若高优先级规则之间冲突，**MUST** 停止生成代码并向用户说明冲突点，不得自行折中。

## 2.1 全局代码标准基线

- `/ai-do` 在执行代码生成与修改时，代码标准 **MUST** 以本文件为准。
- 以下内容沉淀了仓库原有全局开发规则，可直接作为 `/ai-do` 的检查基线。

### 2.1.1 角色与范围

- 你应以高级架构师级别工程师的标准实施代码，重点面向 Java 后端、Spring Boot 与微服务治理场景。
- 目标是产出可直接用于生产环境的企业级代码，而不是示例性或临时性实现。

### 2.1.2 推荐实践

#### 代码风格

- Java 代码 **MUST** 遵循 Google Java Format 的 AOSP（Android Open Source Project）风格。
- 缩进使用 4 个空格。
- 换行后的续行额外缩进 4 个空格。
- 注解一行一个。
- 方法定义的左花括号 `{` 必须与方法签名位于同一行。
- 链式调用需要断行时，应在 `.` 之后换行，并保持续行缩进一致。
- **MUST NOT** 混用其他格式化风格，输出结果必须保持 AOSP 风格一致性。

#### 编码与工具使用

- 空值判断优先使用 `java.util.Objects`。
- 通用工具方法优先使用 Hutool。
- 对象属性复制 **MUST** 使用 `BeanUtil.copyProperties()`。

#### 分层架构

- Controller 层只保留简单校验、参数接收与基于 Java Validation 的参数校验。
- Service / ServiceImpl 层承载核心业务逻辑。
- **MUST** 保持 Controller 薄、Service 厚，避免把业务逻辑堆到接口层。

#### 持久层约定

- 在 MyBatis-Plus 中，查询构造优先使用 `LambdaQueryWrapper`，而不是 `QueryWrapper`，以保证类型安全。

#### 文档与注释

- Controller 与 Service 的方法应补充 Javadoc，至少说明用途、`@param` 与 `@return`。
- ServiceImpl 的关键逻辑节点应添加简洁行内注释，重点说明：
  - 参数校验
  - 分支决策
  - 外部调用
  - 结果组装

### 2.1.3 禁止实践

- **MUST NOT** 使用完全限定类名直接写在代码中，例如 `java.util.List`。
- 应通过 `import` 导入类型，并在代码中使用简短类型名。

## 3. 编码前强制动作

开始实现前，**MUST** 完成以下检查：

- 读取任务文档、设计文档及被修改文件，确认接口、数据结构、边界条件与验收点。
- 读取本文件。
- 识别目标目录中的既有模式：命名、分层、异常处理、日志、测试组织、依赖注入、工具类用法。
- 在开始写代码前，明确列出本次实现需要遵守的“硬性规则清单”。

未完成以上检查，**MUST NOT** 直接生成实现代码。

## 4. 生成代码的硬性要求

### 4.1 一致性

- **MUST** 优先复用项目已有模式，不得引入与现有代码风格明显不一致的新写法。
- **MUST** 保持命名风格一致，包括包名、类名、方法名、变量名、常量名、DTO/VO/Entity 命名。
- **MUST** 保持目录结构一致，不得随意新增层级或跨层放置文件。

### 4.2 最小改动

- **MUST** 仅实现当前任务要求，不得顺手重构无关代码。
- **MUST** 控制改动范围，优先局部修改，避免大面积格式化或无意义重排。
- 若发现顺手修改是完成当前任务的前置条件，**MUST** 在结果中明确说明原因。

### 4.3 可维护性

- **MUST** 让控制流清晰，避免过深嵌套、重复逻辑与含义不明的临时变量。
- **MUST** 做必要的空值、边界、异常或失败分支处理。
- **MUST** 保持接口输入输出语义清晰，避免隐式副作用。
- **MUST NOT** 引入无依据的“未来扩展点”、无使用方的抽象层或过度设计。

### 4.4 依赖与实现方式

- **MUST** 优先使用项目中已存在的库、工具类、基础设施与封装约定。
- **MUST NOT** 因个人偏好替换项目既有实现方案。
- 未经任务或设计要求，**MUST NOT** 新增第三方依赖、基础设施配置或跨模块耦合。

### 4.5 注释与文档

- 仅在逻辑不直观时添加简洁注释，解释“为什么”，不要解释显而易见的“做了什么”。
- 若任务要求同步文档、示例、SQL、配置说明，**MUST** 在本次实现中一并完成，不得遗漏。

## 5. 验证阶段强制检查

完成实现后，**MUST** 按以下顺序验证：

1. 功能是否满足 TASK / DESIGN 中的验收要求
2. 是否违反本规范中的硬性要求
3. 是否破坏现有代码风格、分层或目录约定
4. 是否存在明显未处理的异常路径、边界条件或回归风险

输出完成报告时，**MUST** 单独增加“代码规范检查”小节，明确说明：

- 已对照的规范来源
- 本次遵守的关键规则
- 是否发现冲突或例外

## 6. 冲突处理

- 若任务要求与项目现有写法冲突，默认优先遵循任务与设计，但 **MUST** 在结果中注明偏离点。
- 若本规范与现有代码写法冲突，默认优先遵循任务与设计，再尽量兼容现有风格并把改动限制在当前任务范围。
- 若无法判断哪条规则优先，**MUST** 暂停并向用户确认。
