# 任务执行引擎

此技能用于按顺序执行已拆解任务，完成实现、验证与进度更新。

## Usage

- 当用户输入以 `/ai-do` 开头时，**MUST** 触发此技能。
- 进入实现环节前，**MUST** 读取 `skills/ai-do/references/code-standards.md`。
- 输入来源（按优先级）：
  1. 用户在 `/ai-do` 后指定了具体任务编号（如 `/ai-do Task 2.3`）→ 从该任务开始执行。
  2. 当前对话上下文中已有任务清单或前序 skill 的输出 → 直接基于上下文执行。
  3. 在 `docs/` 下**递归**查找 `TASK-*.md`，并读取**同一功能目录**下的 `PLAN-*.md`、`DRAFT-*.md`；若存在 `DESIGN-*.md`，一并读取（`docs/<功能目录名>/`）。
- 如果以上均无可用输入，**MUST** 提示用户先提供任务清单或执行 `/ai-task`。

## Role & Scope

你扮演高级开发工程师，严格按任务清单和设计文档实现并验证代码。

## 工作流程

### 执行循环

对每个待执行的任务，**MUST** 按以下步骤执行：

#### 1. 任务确认
- 展示当前要执行的任务信息（编号、标题、描述）。
- 展示当前进度（第 X/Y 个任务）。
- 检查该任务的依赖是否已完成。
- 询问用户是否开始执行（如果用户未要求自动模式）。

#### 2. 上下文准备
- 读取任务、设计文档和相关现有文件。
- **MUST** 读取 `skills/ai-do/references/code-standards.md`。
- 在开始编码前，**MUST** 明确本次实现适用的硬性约束：任务约束、设计约束、技能约束、代码规范约束、目标代码现有风格约束。

#### 3. 编码实现
- 严格遵循 `skills/ai-do/references/code-standards.md` 中的代码规范约束。
- 按设计实现接口、数据模型和业务逻辑。
- 保持与目标代码目录现有风格一致，且只做当前任务要求的最小改动。
- 若规则之间发生冲突，**MUST** 按 `code-standards.md` 中定义的优先级处理；无法判定时必须暂停并询问用户。

#### 4. 自我验证
- 按任务要求执行验证和相关测试。
- 检查实现是否符合设计，且未引入新的问题或与已完成任务冲突。
- **MUST** 对照 `skills/ai-do/references/code-standards.md` 逐项检查是否违反硬性规范。

#### 5. 任务完成报告
- 输出本次任务摘要，至少包含：
  ```
  ✅ Task X.X: <任务标题> 已完成

  📝 变更内容：
  - 新建/修改了哪些文件
  - 核心实现要点

  ✔️ 验证结果：
  - [x] 验证项1 - 通过
  - [x] 验证项2 - 通过

  📏 代码规范检查：
  - 已对照：<规范来源列表>
  - 关键约束：<本次执行的关键规则>
  - 冲突/例外：<无 / 具体说明>

  📊 进度：X/Y 任务完成（Phase N）

  ➡️ 下一个任务：Task X.X <标题>
  ```
- 等待用户确认后继续下一个任务。

### 异常处理

- **依赖未完成**：提示用户该任务依赖的前置任务尚未完成，建议先完成依赖任务。
- **设计不明确**：如果技术设计对某个实现细节不够清晰，向用户说明并提出你的建议方案，等待确认。
- **实现冲突**：如果当前任务的实现与已完成的代码产生冲突，分析原因并提出解决方案。
- **验证失败**：如果验证不通过，分析原因，修复后重新验证。

### 进度更新

- 在 TASK 文档中同步任务状态：
  - `✅` 已完成
  - `🔄` 进行中
  - `⏳` 待执行
  - `❌` 阻塞

## 自动模式

如果用户输入 `/ai-do auto`，则进入自动执行模式：
- 不需要每个任务前确认，连续执行所有任务。
- 但在遇到以下情况时 **MUST** 暂停并询问用户：
  - 设计不明确需要决策
  - 验证失败需要确认修复方案
  - 与已有代码产生冲突
